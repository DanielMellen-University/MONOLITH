# .AGENTS.md — Monolith Development Workflow

> This is a hidden file (`.AGENTS.md`) so it does not clutter the project root.

This file defines how AI agents (and human contributors) should work in this repository.

## Branching Strategy

- **`beta`** is the primary development branch.
  - All new work, features, experiments, and refactoring happen here.
  - This is the default branch you should be on for almost all tasks.

- **`main`** is the **stable / "lived-in"** branch.
  - It represents the current version of Monolith that the owner is comfortable running as their daily personal environment.
  - It should only be updated for **big, polished releases**.

## Rules for AI Agents

1. **Always work on `beta`**
   - At the start of any task, check the current branch with `git branch --show-current`.
   - If you are not on `beta`, immediately switch to it:
     ```bash
     git checkout beta
     ```
   - Never make code changes, commits, or pushes while on `main`.

2. **Do not merge to `main` without explicit approval**
   - You must **never** merge `beta` into `main` (or push directly to `main`) on your own initiative.
   - Merging beta → main is reserved for deliberate, polished releases.
   - Only perform a merge to `main` when the user explicitly says something like:
     - "Merge beta to main"
     - "Do a release"
     - "Promote this to stable"
     - "Update main"

3. **Release Process (beta → main)**
   - When the user requests a release:
     - Ensure `beta` is in a good state (clean build, key features feel solid).
     - Create a clear merge commit or use a merge with a good message.
     - Push the update to `main`.
   - After merging, the user may want to tag the release (e.g. `git tag lived-in-YYYY-MM`).

4. **When in doubt**
   - Stay on `beta`.
   - Ask the user before doing anything that affects `main`.

## Additional Notes

- This is a long-term personal project. The goal is deliberate, sustainable growth rather than rapid iteration on `main`.
- `main` should generally feel like a reliable place the owner can "live inside."
- Feature branches off `beta` are allowed for large or risky pieces of work, but they should be merged back into `beta` once complete.

Follow these rules strictly. The stability of the owner's daily environment depends on it.
